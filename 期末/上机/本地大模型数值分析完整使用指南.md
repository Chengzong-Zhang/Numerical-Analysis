# 本地大模型解决数值分析问题：完整使用指南

> 适用环境：Windows、PowerShell、Ollama、RTX 4060 Laptop 8GB  
> 当前主力模型：`qwen35-9b-uncensored:latest`  
> 目标：兼顾复杂数值分析题的推理质量、响应速度和代码可靠性。

## 一、先理解影响回答速度的因素

本地大模型回答一道题通常分为两个阶段：

1. **读取输入阶段（prefill）**：模型读取题目、历史对话、代码和文档。
2. **生成输出阶段（decode）**：模型逐个生成思考过程和最终答案。

影响总时间的因素，按通常的重要程度排列：

| 因素 | 对速度的影响 | 调整建议 |
|---|---:|---|
| 模型参数量 | 很大 | 4B 模型比 9B 模型快，但复杂推导能力通常更弱 |
| 思考模式与思考深度 | 很大 | 普通题使用 `low` 或 `medium`，困难题才使用 `high` |
| 实际输入和对话历史长度 | 很大 | 每道独立题尽量开启新对话，只提供必要材料 |
| 输出长度 | 较大 | 限制 `num_predict`，避免模型无限扩写 |
| 上下文窗口上限 `num_ctx` | 间接影响 | 上限不等于实际使用量，但过大时会增加显存占用 |
| 量化方式和 GPU | 很大 | 8GB 显存适合约 4B 至 9B 的量化模型 |

对于数值分析，不能只追求速度。较可靠的策略是：

```text
简单解释与语法修改：小模型或 Qwen + think low
普通计算题与算法实现：Qwen 9B + think medium
困难证明、稳定性与收敛性推导：Qwen 9B + think high
重要结果：必须运行代码、检查误差或代回公式验证
```

- `UNTIL`：模型还会在内存或显存中保留多久。

```powershell

当前实际上下文 num_ctx：16384
当前最大输出 num_predict：8192

模型理论支持 `262144` tokens，不代表你的 8GB 显卡适合实际启用这么大的上下文。

其中常见参数含义：

PARAMETER temperature 0.7     # 随机性；数值分析建议降低
PARAMETER top_p 0.9           # 控制候选 token 的采样范围
```

## 三、为数值分析选择模型和思考深度

### Qwen 9B

适合作为数值分析主力模型：

- 中文题目理解较好。
- 更适合复杂推导、算法设计和详细中文注释。
- 使用思考模式时正确率通常更高，但响应更慢。

### Gemma

注意模型名称通常是 **Gemma**，不是 Gamma。

Gemma 小模型更适合：

- 快速解释概念。
- 修改 Python 或 MATLAB 语法。
- 生成简单算法框架。
- 对 Qwen 的答案做快速复查。

Gemma 小模型通常响应更快，但不建议仅依靠它完成困难的收敛性证明、稳定性推导或复杂边值问题。

### 推荐分工

| 任务 | 推荐模型 | 思考深度 |
|---|---|---|
| 查询 Python/MATLAB 语法 | Gemma 小模型或 Qwen | `false` / `low` |
| 解释 Euler、RK4、插值等基本概念 | Qwen 9B | `low` |
| 普通数值分析计算题 | Qwen 9B | `medium` |
| 收敛阶、稳定域、误差证明 | Qwen 9B | `high` |
| 阅读较长 PDF 并生成详细程序 | Qwen 9B | `medium` / `high` |

## 四、最常用的启动命令

### 1. 普通数值分析题

```powershell
ollama run qwen35-9b-uncensored:latest --think medium
```

### 2. 特别困难的推导题

```powershell
ollama run qwen35-9b-uncensored:latest --think high
```

只在确实需要长推导时使用 `high`。否则模型可能花费大量时间探索无关路径。

### 3. 快速语法修改或简单解释

```powershell
ollama run qwen35-9b-uncensored:latest --think low
```

### 4. 不需要推理的简单任务

```powershell
ollama run qwen35-9b-uncensored:latest --think false
```

### 5. 查看实际运行速度

```powershell
ollama run qwen35-9b-uncensored:latest --think medium --verbose
```

退出对话后，`--verbose` 会显示：

- 输入 token 数和读取速度。
- 输出 token 数和生成速度。
- 总运行时间。

### 6. 在命令行安装和调用 Gemma

注意：Google 的模型名称是 **Gemma**，不是 Gamma。执行命令时必须使用 Ollama 中显示的准确名称，例如 `gemma3:4b`。

#### 第一步：检查 Gemma 是否已经安装

```powershell
ollama list
```

如果输出中没有 `gemma`，说明本机尚未安装 Gemma。下载模型需要保持网络连接。

#### 第二步：下载适合 8GB 显存的 Gemma

推荐先使用较轻量的 `gemma3:4b`：

```powershell
ollama pull gemma3:4b
```

下载完成后再次检查：

```powershell
ollama list
```

模型名称以 `ollama list` 的实际输出为准。如果安装的是其他 Gemma 版本，后续命令中的 `gemma3:4b` 也要替换成对应名称。

#### 第三步：进入持续对话模式

```powershell
ollama run gemma3:4b
```

出现输入提示符后，可以直接输入问题，例如：

```text
请解释经典四阶 Runge-Kutta 方法，并说明四个阶段斜率的作用。
```

退出持续对话：

```text
/bye
```

#### 第四步：单次提问

下面的命令会完成一次提问，然后自动返回 PowerShell：

```powershell
ollama run gemma3:4b "请解释经典四阶 Runge-Kutta 方法"
```

#### 第五步：选择思考深度并查看速度

快速解释或语法修改：

```powershell
ollama run gemma3:4b --think low --verbose
```

普通数值分析问题：

```powershell
ollama run gemma3:4b --think medium --verbose
```

如果当前 Gemma 版本不支持某个思考深度，Ollama 会报错或退回模型支持的行为。此时可删除 `--think` 参数后重新运行。

#### 第六步：查看运行状态和模型信息

```powershell
ollama ps
ollama show gemma3:4b
```

`ollama ps` 用于检查 Gemma 是否使用 GPU、实际上下文窗口和保留时间；`ollama show` 用于查看模型参数、能力和理论上下文长度。

#### 第七步：停止 Gemma 并释放显存

```powershell
ollama stop gemma3:4b
```

停止模型不会删除模型文件。下次仍可通过 `ollama run gemma3:4b` 启动。

#### 第八步：不再需要时删除 Gemma

```powershell
ollama rm gemma3:4b
```

该命令会删除本地模型文件并释放磁盘空间。删除后若要再次使用，必须重新执行 `ollama pull gemma3:4b`。

#### 第九步：用同一道题比较 Gemma 和 Qwen

分别运行：

```powershell
ollama run gemma3:4b "请推导经典四阶 Runge-Kutta 方法的局部截断误差，并给出 Python 实现" --verbose
ollama run qwen35-9b-uncensored:latest "请推导经典四阶 Runge-Kutta 方法的局部截断误差，并给出 Python 实现" --think medium --verbose
```

比较以下项目：

- 首次输出等待时间。
- `eval rate`，即每秒生成 token 数。
- 数学推导是否正确。
- Python 代码能否直接运行。
- 是否提供误差或收敛阶验证。

Gemma 响应较快时，可用于概念解释和快速复查；复杂数值分析推导仍建议优先使用 Qwen，并通过运行代码验证。

## 五、推荐的三套 Qwen 配置

当前默认配置 `num_ctx=16384`、`num_predict=8192` 能处理长题，但最大输出偏长。建议建立不同用途的模型配置。

### 配置 A：快速问答

适合语法问题、概念查询和短代码修改：

```text
num_ctx：4096
num_predict：1024
temperature：0.2
thinking：low
```

新建 `Modelfile-qwen-fast`：

```text
FROM qwen35-9b-uncensored:latest
PARAMETER num_ctx 4096
PARAMETER num_predict 1024
PARAMETER temperature 0.2
PARAMETER top_p 0.9
```

创建并运行：

```powershell
ollama create qwen-numerical-fast -f .\Modelfile-qwen-fast
ollama run qwen-numerical-fast --think low
```

### 配置 B：日常数值分析，推荐默认使用

适合大多数数值分析题、算法实现和代码讲解：

```text
num_ctx：8192
num_predict：4096
temperature：0.2
thinking：medium
```

新建 `Modelfile-qwen-numerical`：

```text
FROM qwen35-9b-uncensored:latest
PARAMETER num_ctx 8192
PARAMETER num_predict 4096
PARAMETER temperature 0.2
PARAMETER top_p 0.9
```

创建并运行：

```powershell
ollama create qwen-numerical -f .\Modelfile-qwen-numerical
ollama run qwen-numerical --think medium
```

### 配置 C：长 PDF 与困难推导

适合读取较长材料、写报告或完成复杂推导：

```text
num_ctx：16384
num_predict：4096
temperature：0.2
thinking：high
```

你的 8GB 显卡不建议日常使用超过 `16384` 的上下文。更大的上下文会增加显存压力，也可能导致部分计算转移到 CPU，从而明显变慢。

## 六、上下文应该如何使用

### 1. `num_ctx` 不等于当前已经使用的上下文

`num_ctx=16384` 表示最多允许使用约 `16384` tokens，并不表示每次提问都会读取这么多内容。

真正影响输入阶段速度的是：

```text
系统提示词 + 当前题目 + 已有对话历史 + 附加文档 + 需要保留的输出
```

### 2. 如何查看上下文窗口上限

```powershell
ollama ps
ollama show qwen35-9b-uncensored:latest
```

### 3. 如何查看一次 API 请求实际处理的 token 数

在 PowerShell 中调用 Ollama API：

```powershell
$body = @{
    model = "qwen35-9b-uncensored:latest"
    prompt = "请简要解释经典四阶 Runge-Kutta 方法。"
    stream = $false
} | ConvertTo-Json

$result = Invoke-RestMethod `
    -Uri "http://127.0.0.1:11434/api/generate" `
    -Method Post `
    -ContentType "application/json" `
    -Body $body

$result | Select-Object prompt_eval_count, eval_count, total_duration
```

字段含义：

- `prompt_eval_count`：本次输入实际使用的 token 数。
- `eval_count`：本次生成的 token 数。
- `total_duration`：本次请求总耗时，单位通常为纳秒。

### 4. 控制上下文的实际做法

- 每道独立题开启一个新对话。
- 不要把整本教材都放入上下文，只提供相关章节。
- 先让模型读取题目，再按需补充公式或代码。
- 修改代码时提供报错和相关函数，不要反复粘贴全部项目。
- 长 PDF 可以按章节或题目拆分处理。

## 七、解决数值分析题的可靠 Workflow

完整提示词拆题流程也可参阅：

```text
期末\上机\03_Python算法模板\本地大模型提示词workflow.md
```

### 第一步：先识别题型，不要立即写代码

发送：

```text
下面是一道数值分析题。

题目：
[粘贴完整题目]

请先不要写代码。请完成：
1. 判断题目属于哪类数值方法。
2. 列出已知量、未知量、输入数据和目标输出。
3. 写出需要使用的核心公式。
4. 说明算法步骤、停止条件和误差估计方法。
5. 指出最容易出现的数学错误。
6. 检查题目给出的数据是否相互一致。

只进行必要推导，不要扩展无关内容。
```

检查模型对题目的理解正确后，再进入下一步。

### 第二步：要求给出伪代码和验证方案

```text
基于刚才确认的数学推导，请先不要写完整程序。

请给出：
1. 算法伪代码。
2. 每个函数的输入、输出和职责。
3. 数据更新顺序。
4. 误差或收敛阶的验证方式。
5. 可以用于人工检查的简单测试案例。
```

### 第三步：生成可运行代码

```text
现在请根据已确认的推导和伪代码生成完整 Python 程序。

要求：
1. 标出所有需要由我替换的数据。
2. 每个核心公式旁写明对应的数值分析意义。
3. 不要依赖未定义的全局变量。
4. 检查数组长度、循环边界和步长是否整除区间。
5. 输出数值结果、参考值、绝对误差和收敛阶。
6. 提供能够直接运行的验证代码。
```

### 第四步：运行并让模型修复具体错误

程序报错时，不要只说“代码不工作”。发送完整报错：

```text
程序运行报错如下：

[粘贴完整错误信息]

请完成：
1. 指出根本原因和对应代码行。
2. 只修改出错部分，不要重写无关代码。
3. 解释修复为什么正确。
4. 给出修复后应观察到的结果。
```

### 第五步：验证数值结果，而不是只验证程序能运行

```text
程序能够运行，但请继续检查数值正确性：

1. 把结果代回原方程或原条件。
2. 与真解、高精度参考值或另一种可靠算法比较。
3. 使用多组步长计算误差和观测收敛阶。
4. 检查结果是否符合方法的理论阶与稳定性结论。
5. 如果不符合，请判断是代码错误、舍入误差还是尚未进入渐近区间。
```

## 八、数值分析专用提示词模板

### 1. 常微分方程

```text
请先识别这是初值问题还是边值问题。若为高阶方程，请先正确化为一阶方程组。

写代码时必须：
- 明确状态向量每个分量的含义。
- 先计算本步所有阶段量，再更新状态。
- 检查步长能否恰好到达终点。
- 若有真解，输出端点误差和多步长观测收敛阶。
- 若无真解，使用加密网格或高阶方法作为参考解。
```

### 2. 数值积分

```text
请先写出求积节点、权重和区间映射公式。

写代码时必须：
- 节点和权重使用数组保存。
- 检查从标准区间到 [a,b] 的映射及缩放因子。
- 对自适应算法，先估计误差，再决定接受还是细分。
- 输出函数调用次数、计算结果和误差估计。
```

### 3. 插值与样条

```text
请区分插值多项式、分段插值和样条。

写代码时必须：
- 检查节点是否互异并按需要排序。
- 明确边界条件属于自然、夹持还是其他类型。
- 用原节点检查插值条件。
- 在测试点与高精度参考值比较。
```

### 4. 线性方程组与迭代法

```text
请先检查矩阵性质，并说明所选算法的适用条件。

写代码时必须：
- 区分残差 ||b-Ax|| 和真实误差 ||x-x*||。
- 明确停止条件、最大迭代次数和初始向量。
- 对迭代法检查收敛条件或谱半径。
- 输出每次迭代的残差变化。
```

### 5. 误差、稳定性和收敛阶

```text
请严格区分局部截断误差、整体误差、绝对稳定性和零稳定性。

推导时必须：
- 写明使用的误差模型。
- 说明阶数结论成立所需条件。
- 用多组步长计算观测阶。
- 解释理论阶与实验阶不一致的原因。
```

## 九、如何在速度与正确率之间切换

### 场景 A：回答明显过慢

依次尝试：

1. 把 `--think high` 改成 `--think medium`。
2. 开启新对话，移除无关历史。
3. 把长题拆成“推导、代码、验证”三个阶段。
4. 使用 `num_ctx=8192` 的 `qwen-numerical`。
5. 限制回答范围，例如要求“只分析算法，不写代码”。

### 场景 B：回答很快但错误较多

依次尝试：

1. 把 `--think low` 改成 `--think medium`。
2. 要求模型先推导，再写代码。
3. 要求输出误差、残差或观测收敛阶。
4. 提供题目中的真解、边界条件和理论阶供其交叉检查。
5. 对特别困难的问题使用 `--think high`。

### 场景 C：对话越聊越慢

这通常是历史上下文不断增长造成的。处理方式：

1. 让模型总结已经确认的结论。
2. 开启新对话。
3. 只粘贴总结、当前代码和最新错误。

## 十、常见误区

### 误区 1：上下文越大，模型越聪明

上下文窗口只决定能够读取多少内容，不直接提高推理能力。无关内容过多反而会降低速度，并可能干扰模型判断。

### 误区 2：思考时间越长，结果一定越正确

长思考可以帮助困难题，但也可能产生无关推导。提示词应明确要求：

```text
只进行解决本题所需的推导；完成后必须用代入、误差或数值实验验证。
```

### 误区 3：程序能够运行就说明算法正确

数值程序必须至少进行一种数学验证：

- 与真解比较。
- 检查残差。
- 检查边界条件。
- 检查守恒量。
- 检查多步长观测收敛阶。
- 与另一种可靠算法比较。

### 误区 4：小模型思考更久就一定能替代大模型

思考时间不能完全弥补模型能力差异。复杂推导优先使用 Qwen 9B；Gemma 小模型更适合作为快速辅助。

## 十一、推荐日常使用流程

每次处理数值分析题时，按下面顺序操作：

1. 启动日常数值分析模型：

   ```powershell
   ollama run qwen-numerical --think medium --verbose
   ```

2. 粘贴题目，让模型先识别题型并推导，不要立即写代码。
3. 人工检查核心公式、边界条件和停止条件。
4. 让模型给出伪代码和验证方案。
5. 让模型生成完整代码，并标记需要替换的数据。
6. 在本机运行代码，把完整错误或结果交给模型检查。
7. 使用真解、残差或收敛阶验证。
8. 完成一道独立题后开启新对话，避免历史上下文不断增长。

推荐默认参数：

```text
模型：Qwen 9B
上下文：8192
最大输出：4096
思考深度：medium
temperature：0.2
```

只有读取长 PDF 或处理复杂证明时，才切换到：

```text
上下文：16384
思考深度：high
```

## 十二、快速命令速查

```powershell
# 查看本地模型
ollama list

# 查看正在运行的模型和实际上下文窗口
ollama ps

# 查看模型参数
ollama show qwen35-9b-uncensored:latest

# 查看完整 Modelfile
ollama show qwen35-9b-uncensored:latest --modelfile

# 普通数值分析题
ollama run qwen35-9b-uncensored:latest --think medium --verbose

# 困难推导题
ollama run qwen35-9b-uncensored:latest --think high --verbose

# 快速简单任务
ollama run qwen35-9b-uncensored:latest --think low --verbose

# 下载 Gemma 3 4B，需要网络连接
ollama pull gemma3:4b

# 启动 Gemma 持续对话
ollama run gemma3:4b

# 使用 Gemma 快速回答并显示速度
ollama run gemma3:4b --think low --verbose

# 查看 Gemma 模型信息
ollama show gemma3:4b

# 停止 Gemma 并释放显存
ollama stop gemma3:4b

# 删除 Gemma 本地模型文件，执行前确认不再需要
ollama rm gemma3:4b

# 检查 Ollama API
Invoke-RestMethod http://127.0.0.1:11434/api/tags
```
